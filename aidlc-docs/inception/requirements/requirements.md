# 要件定義書

## 1. 意図分析サマリー

| 項目 | 内容 |
|------|------|
| ユーザーリクエスト | 統合業務申請・承認プラットフォームの構築 |
| リクエストタイプ | 新規プロジェクト（グリーンフィールド） |
| スコープ | MVP — 経費精算業務のみを先行実装 |
| 複雑度 | 中程度（MVP範囲に限定したことで複雑度を抑制） |
| 将来展望 | 出張・購買・稟議・備品管理は後続フェーズで対応 |

## 2. プロジェクト概要

### 2.1 システム名（案）
統合業務申請・承認プラットフォーム（MVP: 経費精算モジュール）

### 2.2 目的
社内の経費精算業務をデジタル化し、申請・承認プロセスの効率化とガバナンス強化を実現する。将来的に他の申請業務（出張・購買・稟議・備品管理）へ拡張可能な基盤を構築する。

### 2.3 対象規模

| 利用者区分 | 人数規模 | MVP時の主な役割 |
|-----------|---------|----------------|
| 一般社員 | 約1,000名 | 経費精算申請の起票、申請状況の確認 |
| 管理職 | 約120名 | 経費精算申請の承認・差戻し |
| 管理部門（経理） | — | 経費精算内容の確認・処理 |
| システム管理者 | 約10名 | 承認ルート設定、権限管理、システム運用 |

## 3. 技術スタック

| レイヤー | 技術 |
|---------|------|
| フロントエンド | React（TypeScript） |
| バックエンド | Python（FastAPI） |
| データベース | PostgreSQL |

## 4. 機能要件

### FR-001: 経費精算申請
- 交通費・交際費・消耗品費等の経費精算申請を起票できること
- 申請項目: 申請日、経費種別、金額、発生日、摘要、添付ファイル（領収書等）
- 複数明細行を1申請にまとめて起票できること
- 下書き保存ができること
- 申請の取消し（承認前）ができること

### FR-002: 承認ワークフロー
- 承認ルートは固定2段階（直属上長 → 部門長）とする
- 申請種別・金額に応じた承認ルートの自動判定は将来対応とする
- 承認者は申請内容を確認し、承認または差戻しができること
- 差戻し時はコメントを付与できること
- 差戻された申請は修正して再申請できること

### FR-003: 代理申請・代理承認
- 代理申請: 他の社員に代わって経費精算申請を起票できること
- 代理承認: 承認者不在時に指定された代理者が承認できること
- 代理設定はシステム管理者が事前に登録すること

### FR-004: 添付ファイル管理
- 申請に領収書等のファイルを添付できること
- 対応ファイル形式: PDF、JPEG、PNG
- 容量制限: 1ファイルあたり5MB、1申請あたり合計10MB
- アップロード済みファイルのプレビュー・ダウンロードができること

### FR-005: 申請状況照会
- 申請者は自分の申請一覧と各申請の進捗状況を確認できること
- 承認者は自分に割り当てられた承認待ち一覧を確認できること
- 検索条件: 申請日範囲、経費種別、ステータス、金額範囲
- ステータス: 下書き、申請中、承認待ち（上長）、承認待ち（部門長）、承認済み、差戻し、取消し

### FR-006: 管理者向けダッシュボード
- 経費精算の申請件数・処理状況・滞留状況を集計表示できること
- 期間別（日次・週次・月次）の集計ができること
- 部門別の集計ができること
- 滞留申請（一定期間未処理）のアラート表示ができること

### FR-007: メール通知
- 以下のタイミングでメール通知を送信すること:
  - 承認者への承認依頼通知
  - 申請者への承認完了通知
  - 申請者への差戻し通知
  - 代理承認者への承認依頼通知
- 通知メールから申請詳細画面へ直接遷移できること

### FR-008: ユーザー管理
- ユーザーの登録・編集・無効化ができること
- ロール（一般社員・管理職・経理担当・システム管理者）の割り当てができること
- 所属部門・直属上長・部門長の設定ができること

### FR-009: 承認ルート設定
- 部門ごとの承認ルート（直属上長 → 部門長）を設定できること
- 承認ルートの有効期間を設定できること（組織改編対応）

### FR-010: 監査ログ
- すべての操作（申請・承認・差戻し・設定変更等）の監査ログを記録すること
- 監査ログの検索・閲覧ができること（管理者・経理担当）
- ログ項目: 操作日時、操作者、操作種別、対象データ、変更前後の値

### FR-011: 外部連携（バッチ）
- 人事マスタ連携: 社員情報・所属部門・役職をバッチ取込できること
- 組織マスタ連携: 組織階層情報をバッチ取込できること
- 会計システム連携: 承認済み経費精算データを仕訳データとしてバッチ出力できること
- バッチ実行スケジュールは設定可能とすること

### FR-012: 多言語対応
- 画面表示言語として日本語・英語を切り替えられること
- ユーザーごとに表示言語を設定できること
- 申請データ自体は入力言語のまま保存すること

## 5. 非機能要件

### NFR-001: 可用性
- 平日業務時間帯（9:00〜18:00 JST）の安定稼働を保証すること
- 計画メンテナンスは業務時間外に実施すること
- 月間稼働率: 99.5%以上

### NFR-002: 性能
- 同時利用ユーザー数: 100ユーザー程度
- 画面レスポンスタイム: 3秒以内（95パーセンタイル）
- バッチ処理: 1,000件の経費精算データを30分以内に処理
- API レスポンスタイム: 1秒以内（95パーセンタイル）

### NFR-003: セキュリティ
- SAML / OIDC による認証連携（外部IdP）
- ロールベースアクセス制御（RBAC）
- 個人情報保護: 申請データへのアクセスは権限に基づき制限
- 操作ログ・監査ログの記録と保管
- 通信の暗号化（TLS 1.2以上）
- データの暗号化（保存時・転送時）
- セキュリティ拡張ルール（SECURITY-01〜15）を全面適用

### NFR-004: 信頼性
- RTO（目標復旧時間）: 4時間以内
- RPO（目標復旧時点）: 1時間以内
- 障害時の復旧手順を文書化すること
- データベースの定期バックアップ（1時間ごと）

### NFR-005: 保守性
- 組織改編時の承認ルート変更が管理画面から実施できること
- マスタデータのバッチ取込による一括更新が可能であること
- 将来の業務拡張（出張・購買・稟議・備品管理）に対応可能なアーキテクチャとすること

### NFR-006: データ管理
- 申請データの保管期間: 3年間
- 保管期間経過後のデータアーカイブ・削除ポリシーを定義すること
- 添付ファイルの容量制限: 1ファイル5MB、1申請合計10MB
- 添付ファイルの保管先: Amazon S3（暗号化有効）

### NFR-007: 対応環境
- 対応ブラウザ: Google Chrome（最新版）、Microsoft Edge（最新版）
- PC利用のみ（モバイル対応は将来検討）
- 画面解像度: 1280×720以上

## 6. 制約事項

- MVP範囲は経費精算業務のみとし、他業務は後続フェーズで対応する
- 承認ルートはMVPでは固定2段階（直属上長 → 部門長）とする
- 通知はMVPではメール通知のみとし、チャット通知は後続対応とする
- 外部連携はバッチ方式とし、リアルタイム連携は将来検討とする

## 7. 前提条件

- 外部IdP（Azure AD / Okta等）が利用可能であること
- AWSアカウントが利用可能であること
- 人事マスタ・組織マスタのデータフォーマットが提供されること
- 会計システムの仕訳データフォーマットが提供されること
- メール送信用のドメインが利用可能であること（Amazon SES）

## 8. 用語定義

| 用語 | 定義 |
|------|------|
| 経費精算 | 業務上発生した費用の精算を申請する業務 |
| 承認ルート | 申請が承認されるまでに経由する承認者の順序 |
| 直属上長 | 申請者の直接の上司 |
| 部門長 | 申請者が所属する部門の責任者 |
| 差戻し | 承認者が申請内容に不備がある場合に申請者に返却すること |
| 代理申請 | 他の社員に代わって申請を起票すること |
| 代理承認 | 承認者不在時に指定された代理者が承認を行うこと |
| MVP | Minimum Viable Product（最小限の実用的な製品） |
